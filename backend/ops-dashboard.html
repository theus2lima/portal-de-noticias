📖 GUIA DE USO — OPS DASHBOARD

Início Rápido

1. Conectar ao backend

Preencha os dois campos no topo da página e clique em Connect:

CampoO que colocar
BACKEND URLURL do servidor Node.js (ex: http://localhost:3001 ou https://seu-backend.railway.app)
OPS KEYValor da variável OPS_SECRET do seu .env
Dica: você pode passar URL e chave via URL para salvar nos favoritos:
ops-dashboard.html?url=https://...&key=sua-chave

2. Confirmar conexão

  • O indicador SSE no topo fica verde — stream de logs conectado
  • O card Overall mostra 🟢 OK, 🟡 DEGRADED ou 🔴 CRITICAL
  • As métricas começam a ser atualizadas a cada 2 segundos

3. O que fazer se aparecer OFFLINE

  • Verifique se o servidor está rodando (npm start na pasta backend/)
  • Confirme que a URL não tem barra no final
  • Confirme que o OPS_SECRET está correto

Interpretando o Status

Status geral (Overall)

StatusO que significaAção
OK Tudo saudável — Redis UP, DB respondendo, Circuit CLOSED Monitorar normalmente

CRITICAL Redis DOWN + DB falhando ou Circuit OPEN simultâneo Intervenção imediata — veja aba Emergências

Métricas em Tempo Real

Todas as métricas são calculadas numa janela deslizante de 60 segundos.

MétricaNormalAlerta 🟡Crítico 🔴
Cache Hit %> 95%< 90%< 85%
p99 Latência< 200ms> 500ms> 1000ms
Error Rate %< 0.5%> 1%> 5%
DB / sBaixo e estávelSubindo enquanto req/s estávelAlto + circuit OPEN

Sinais de alerta a observar

  • Cache Hit caindo com DB/s subindo → TTL muito curto ou Redis com problemas
  • p99 alto mas p95 normal → consultas específicas lentas, verificar índices
  • Error Rate subindo gradualmente → banco sobrecarregado, considerar DEGRADED mode
  • RL Hits crescendo → possível ataque ou scraper, considerar STRICT mode
Os cards ficam com borda colorida automaticamente quando ultrapassam os thresholds de alerta.

Controles Operacionais

Modo do Sistema

ModoComportamentoQuando usar
NORMAL Operação padrão — cache + banco normais Sempre que possível
CACHE_ONLY Ignora o banco completamente. Serve só o que está no Redis. Se não tiver cache, retorna lista vazia Banco instável, manutenção, pico extremo
DEGRADED Timeout do banco reduzido para 2s (vs 5s normal). Fallbacks mais agressivos Banco lento mas ainda respondendo

Circuit Breaker

OpçãoComportamento
AUTO O breaker decide sozinho — abre após 5 falhas consecutivas, tenta reabrir após 30s. Recomendado.
FORCE_OPEN Bloqueia 100% das queries ao banco imediatamente. Zero pressão no DB. Use para isolar o banco manualmente.
FORCE_CLOSE Ignora o estado do breaker e força chamadas ao banco. Cuidado: use só para testar recuperação.

Rate Limit

ModoLimite efetivoQuando usar
NORMALValor do env (padrão 120/min)Operação normal
STRICT20% do normal (≈ 24/min)Pico de tráfego, possível ataque
DISABLEDSem limiteSomente para desblocar falso positivo
DISABLED remove toda proteção. Reative assim que resolver o problema.

Cache

  • Reaquecer Cache — executa o warming (page 1 + top categorias). Use após limpar ou após deploy
  • Limpar Tudo — faz FLUSHDB no Redis. Aumenta carga no banco temporariamente
  • Purgar Chave — invalida uma entrada específica. Formato: news:list:all:1:20 ou news:item:uuid

Bloquear IP

Bloqueia um IP específico — retorna 403 imediatamente, antes de qualquer processamento. Útil para scrapers ou ataques direcionados. O bloqueio é em memória: é removido ao reiniciar o servidor.

Ações Emergenciais

🔥 Survival Mode

Ativa CACHE_ONLY + STRICT rate limit em um clique. Use quando o banco está lento e o tráfego está alto.

  • Bloqueia queries ao banco
  • Reduz o rate limit para 20%
  • Usuários recebem dados do cache (podem estar até 5min desatualizados)
  • Clique de novo para desativar e voltar ao NORMAL
🔄 Reaquecer Cache
  • Popula page 1 da listagem + top categorias no Redis
  • Execute após limpar cache ou após um deploy frio
  • Roda em background — não bloqueia requests
🛡️ Isolar Banco
  • Combina CACHE_ONLY + FORCE_OPEN
  • Zero queries chegam ao Supabase
  • Use quando o banco está completamente fora e você quer evitar cascata de erros
  • Para voltar: clique em NORMAL e AUTO no circuit breaker
🚪 Liberar Rate Limit
  • Desativa o limite por IP
  • Use somente se usuários legítimos estão sendo bloqueados por falso positivo
  • Reative o NORMAL imediatamente após resolver
↺ Reset para NORMAL
  • Volta mode → NORMAL, circuit → AUTO, rate limit → NORMAL
  • Não afeta IPs bloqueados nem cache
  • Use ao finalizar uma intervenção de emergência

Cenários de Falha e Resposta

Redis caiu

DEGRADED O sistema funciona, mas cada request vai ao banco.

  1. Status → Redis: DOWN
  2. Cache hit rate vai para 0% — DB/s vai subir muito
  3. Se o banco aguentar → espere o Redis se recuperar
  4. Se o banco começar a travar → ative DEGRADED mode para timeouts mais curtos
  5. Quando Redis voltar → execute Reaquecer Cache
Banco lento / travado

CRITICAL Queries demoram mais que 5s e começam a falhar.

  1. p99 sobe → Error Rate sobe → Circuit Breaker abre automaticamente
  2. Com o breaker OPEN, sistema serve só cache por 30s
  3. Se o banco não se recuperar em 30s → ative Survival Mode
  4. Opcional: use FORCE_OPEN para garantir zero queries enquanto investiga
  5. Quando banco recuperar → Reset para NORMAL
Pico repentino de tráfego

DEGRADED Req/s sobe abruptamente.

  1. Verifique cache hit rate — se estiver alto (> 90%), o sistema está absorvendo bem
  2. Se DB/s subir junto → cache falhando — execute Reaquecer Cache
  3. Se RL Hits subir → possível ataque — ative STRICT rate limit
  4. Se banco começar a travar → ative Survival Mode
Circuit Breaker preso em OPEN

O breaker tenta reabrir a cada 30s. Se o banco ainda estiver falhando, ele volta para OPEN.

  1. Verifique o banco diretamente no Supabase dashboard
  2. Se banco está OK mas breaker não fecha → use FORCE_CLOSE para forçar
  3. Se banco ainda lento → mantenha em OPEN e aguarde

API Reference

Todos os endpoints requerem o header x-ops-key: <OPS_SECRET>.

Status e Métricas

GET /ops/status → saúde geral + métricas resumidas GET /metrics → snapshot completo (counters, latência, process) GET /health → ping rápido (sem auth)

Controles de Modo

POST /ops/mode { "mode": "NORMAL" | "CACHE_ONLY" | "DEGRADED" } POST /ops/circuit { "action": "AUTO" | "FORCE_OPEN" | "FORCE_CLOSE" } POST /ops/rate-limit { "mode": "NORMAL" | "STRICT" | "DISABLED" } POST /ops/survival { "active": true | false }

Cache

POST /ops/cache/purge { } → limpa TODO o Redis (FLUSHDB) { "key": "news:list:all:1:20" } → purga chave específica POST /ops/cache/refresh → dispara cache warming (async)

IPs

POST /ops/block-ip { "ip": "1.2.3.4", "action": "block" | "unblock" }

Logs

GET /ops/logs?limit=100&level=ERROR → últimas N entradas GET /ops/logs/stream → SSE stream em tempo real

Formato dos logs

{ "ts": 1714235060000, "time": "2025-04-27T19:32:00.000Z", "level": "ERROR" | "WARN" | "INFO" | "DEBUG", "message": "DB timeout: getNewsList exceeded 5000ms" }
